# sciClaw v0.1.71 Announcement

## Tweet 1 - Main Announcement

sciClaw v0.1.71 is out.

This release focuses on safe upgrade behavior for existing Discord routing users:
- Detect old config mismatches after upgrade
- Offer one-step repair in interactive runs
- Add non-interactive `doctor --fix` remediation for team rollouts

https://github.com/drpedapati/sciclaw/releases/tag/v0.1.71

---

## Tweet 2 - What Problem This Fixes

Some users had older routing configs where one or more of these stayed in place:
- `require_mention=false` on routed Discord mappings
- `routing.unmapped_behavior=default`
- empty `channels.discord.allow_from`

That combination can make the bot look over-responsive in shared channels, especially after install/upgrade changes.

v0.1.71 adds targeted config health checks so upgrades are safer by default.

---

## Tweet 3 - Interactive Repair

When you run `sciclaw app` (or `gateway`, `service`, `status`, `agent`) in an interactive terminal, sciClaw now checks for known config mismatches and asks yes/no before applying fixes.

If you accept:
- it creates a config backup
- applies only selected fixes
- writes a routing reload trigger

So users keep control, but do not need to hand-edit JSON.

---

## Tweet 4 - Fleet Repair with Doctor

`sciclaw doctor --fix` now includes config health migration logic for these Discord routing cases.

It is designed for repeated use:
- first run applies needed changes and reports what changed
- second run reports no further config health fixes needed

This is useful for admins supporting multiple machines after Homebrew upgrades.

---

## Tweet 5 - Upgrade and Verify

For existing installs:

```bash
brew update && brew upgrade sciclaw
sciclaw doctor --fix
sciclaw doctor
```

For new installs:

```bash
brew tap drpedapati/tap && brew install sciclaw
sciclaw app
```

Docs: https://sciclaw.dev/docs.html
